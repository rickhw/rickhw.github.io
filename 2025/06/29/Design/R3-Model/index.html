R3 Model

C4 model 是一個 2018 年之後發展出來的 架構描述方式，我在 2017 年底也在組織內部弄了類似概念的表達架構方式，很巧不巧，概念很 C4 model 很像，這段想法在我的著作 SRE實踐與開發平台指南 有提到，底下用 FB 寫的草稿，讓 ChatGPT 幫我整理的想法～

R3 Model：以角色職責為核心的系統架構思維

在與朋友討論系統架構設計時，我經常提到自己多年來實務上使用的一套方法論，它與大家熟悉的 C4 model 有些類似，但核心關注點其實是從「組織職責」與「封裝邊界」出發，甚至偷偷塞了點康威定律進去。

這個方法論沒有什麼炫砲的名字，乾脆就叫它 R3 Model 吧：分為三個層次，對應 High Level View、Logical View、Physical View。R3，簡單粗暴，就代表三層。

🧠 背後思維：OOP、封裝與 R&R

這整套架構方法的邏輯起點來自 OOP（物件導向）與封裝概念。我把一個 Role 想像成一個 Interface，而其內部的邏輯與協作，就像是 implementation。而這個 Interface，最重要的是它代表「誰該做什麼」——也就是 Role & Responsibility（R&R）。

許多技術選型的討論（例如 API Gateway vs Load Balancer vs Reverse Proxy），看似在談技術，其實本質上是在問一個問題：它在這個系統裡的角色與責任是什麼？

🔭 第一層：High Level View（對齊觀點的入口）

這層概念對應到 C4 model 的 Context，用來給業務單位、老闆、非技術利害關係人快速對齊共識。主要涵蓋：

  • 使用者與系統互動方式
  • 內外部系統的整合關係
  • 系統之間的 ACL（Access Control）

我這邊加入了 ACL 的設計，來呈現「可存取性與依賴性」，類似於 OOP 中的 public/protected/private 關鍵字，用來表達系統邊界與封裝程度。

✅ 適合：展示整體策略面與系統邊界，快速建立系統存在的價值與關係。

🧩 第二層：Logical View（Service Definition）

這層對應到 C4 的 Container，但我更關注的是「系統內的職責劃分」。我們從這層開始拉出具體的 Role，例如：

  • Web / Console / API Gateway / Database / Cache / Queue 等
  • 每個 Role 的職責（Responsibility）與協作關係
  • 系統間的資料流（Data Flow）：密度（吞吐量）與溫度（頻率）

此外，我會搭配 User Story / Scenario 來解釋整體流程，也會引用 Scrum Team 的比喻：一個團隊需要不同角色合作，架構亦然。

🎵 像是一個樂團需要歌手、吉他手、鼓手、鍵盤手，每個 Role 多少會決定整體架構的「複雜度」。

這層不是要講 implementation，而是要對齊理解，讓懂技術和不懂技術的人都能站在同一張地圖上看系統。

🚀 第三層：Physical View（Go Live 構面）

這層我稱為 Physical View，但其實不是 C4 的 Component Layer，而是更貼近 Deployment Architecture 的思維，關注：

  • 每個 Role 在 runtime 的部署副本（對應到 Kubernetes 的 Pod/Container）
  • 實際的實作技術：Node.js / Spring Boot / FastAPI / MySQL / Redis / RabbitMQ …
  • 每個實體角色如何落地、如何部署、怎麼運維
  • 成本、可觀測性（O11y）、高可用性設計（HA）

這層的誕生也是因為我當時正需要設計一個可觀測系統，沒有這層就無法落實 Metrics、Logging、Tracing。

🎯 若 Logical View 是系統的藍圖，那 Physical View 就是建築工地現場：誰在哪裡、在做什麼、怎麼做到、會不會倒。

🔍 關於 C4 的第四層 Code

C4 有第四層是 Code Level，但我的模型並沒有這一層。主要是因為我的思考起點是從 Infra / DevOps 面切入，而非純開發流程。不過：

搭配 UML 的 Sequence Diagram / State Machine，其實已經足以補足實作邏輯。

🧱 小結：命名為 R3 Model 的小堅持

我並不是從一開始就知道 C4 model。這套方法是我從 2017 年開始，在跑了十幾二十個系統架構實務案之後，逐步抽象與歸納出來的。後來才發現 C4 與我某些層次的定義幾乎一致，也就不打不相識。

這個模型我沒有對外正式發表過名字，只在自己整理的書裡稱呼它為：

  1. High Level View（策略與邊界）
  2. Service Definition（系統角色與任務）
  3. Go Live（部署與運維視角）

但說真的，這世界太需要簡潔一點的術語了，那就叫它 R3 Model 吧，反正只剩我自己在用，命名權還在我手上 😆

如果你也正在設計一個跨團隊、多角色、高複雜度的系統架構，或許可以試試用 R3 Model 當起點，從角色與責任的角度開始畫起。
